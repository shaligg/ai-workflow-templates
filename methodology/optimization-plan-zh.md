# 工作流优化方案（防漂移 / 提质量 / 降 Token）

## 1. 背景与初步结论

基于近期 `wild-server2` 的实际使用记录（Claude + Codex）与模板仓库现状，可归纳为：

- 当前流程主线有效：`wf-init` -> 文档产出 -> 按任务实现。
- 主要风险不是“能力不够”，而是“规则漂移 + 冗余输入 + 搜索范围过大”。
- 优化方向应优先做边界收敛（禁止项），再做自动化增强（能力项）。

---

## 2. 优化目标

1. 减少漂移：同一规则只维护一份，不在多文件重复定义。
2. 增加质量：把验证放到明确关口，避免“已完成但未统一校验”。
3. 降低 Token：减少重复上下文、限制无边界搜索、压缩文档冗余。

---

## 3. 设计原则（建议作为长期约束）

- 禁止优先于功能：先限制错误行为，再增加新能力。
- 单一真源：规则定义只放在 `RULES.md`（及全局规则），其他文件只引用。
- 默认保守执行：默认一次一个任务，只有明确指令才“连续完成全部任务”。

---

## 4. 建议改动（按优先级）

## P0（立即，低风险高收益）

1. 统一“文档预算”为软约束（不硬拦截）
- 规则文本建议：
  - `requirements/design/tasks 以精简为目标，不设硬性字数拦截；若内容偏长，输出可执行版本后附压缩摘要（关键决策/边界/验收）。`

2. 调整验证策略为“一轮统一验证”
- 规则文本建议：
  - `默认完成一轮任务后统一验证（测试/静态检查/关键路径自检）；失败即停止并报告。仅在用户明确要求时改为每 task 验证。`

3. 收紧搜索边界，避免无效全仓扫描
- 规则文本建议：
  - `优先使用 Reference code paths + rg 定点检索；避免 find -exec 全仓扫描，除非用户明确要求。`

4. 明确“默认单任务模式”
- 保持现有规则，但在提示模板中重复一次，减少执行歧义。

## P1（短期，稳定性增强）

1. Feature 文档模板最小化
- `requirements.md/design.md/tasks.md` 只保留“执行所需字段”，去掉长篇说明。
- 将说明性内容放入 `methodology/`，避免每次任务重复注入。

2. 增加“引用完整性”检查
- 检查模板内是否引用过时路径或旧文档名，避免结构改后指向仍是旧位置。

3. 增加“规则重复度”检查
- 对 `global-rules`、`templates/workflows/l2`、`templates/workflows/l3` 做关键段落重复检测。

## P2（可选）

1. 自动任务循环（可选模式）
- 保留为 Optional，不作为默认模式。
- 默认仍人工驱动，避免失控和超改。

2. Skill 化封装
- 将高频动作（初始化后提示模板、需求提炼模板）做成 skill 文档，降低重复 prompt 成本。

---

## 5. 推荐修改位置（文件级）

建议优先修改以下文件：

1. `global-rules/00-global.md`
- 增补三条全局边界：
  - 文档预算软约束
  - 一轮统一验证
  - 定点搜索优先

2. `templates/workflows/l2/RULES.md`
- 与全局规则保持一致，不重复定义冲突版本。
- 保留 L2 特有约束（任务粒度、需求提炼）。

3. `templates/workflows/l3/RULES.md`
- 仅保留 L3 特有内容（架构一致性、风险与迁移）。
- 不重复全局基础规则。

4. `templates/workflows/l2/features/_template/*.md`
- 保持模板短小，强化可执行字段。

---

## 6. 验收标准（用于判断优化是否成功）

1. 漂移
- 同一规则不在 3 个以上文件重复出现。
- L2/L3 规则仅保留“本级差异”，全局共性不重复。

2. 质量
- 每轮任务结束都有统一验证记录（通过或失败原因）。
- 失败能快速定位到具体任务/文件。

3. Token
- 单次实现前的必读上下文稳定在“核心规则 + 当前 feature 三文档”。
- 需求讨论轮次和“重复解释轮次”明显下降。

---

## 7. 最小落地顺序（推荐）

1. 先改规则文本（全局 + L2 + L3）。
2. 再瘦身 feature 模板（requirements/design/tasks）。
3. 最后补 CI 检查（重复/引用/模板漂移）。

该顺序改动小、回滚简单、收益最快。
